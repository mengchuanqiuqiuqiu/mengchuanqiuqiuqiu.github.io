Diff算法

发表于 2022-02-20 | 分类于 React

diff 作为 Virtual Dom 的加速器，其算法上的改进优化是 React 整个界面渲染的基础和性能保障。React 最值得称道的部分莫过于 diff 和 Virtual Dom 的结合,让用户无需关注 Virtual Dom 背后的运行原理。

传统 diff 算法

传统 diff 是通过循环递归来对节点进行对比，效率低下。其算法复杂度可达 n^3，其中 n 为树的节点总数。这种指数的性能对前端渲染场景来说代价过高。如果 React 只是单纯的引进 diff 算法，其效率远远无法满足前端渲染的需求。因此需要一个高效稳定的 diff 算法。

详解 diff

React 通过大胆的策略，将 O(n^3)复杂度的问题转为 O(n)复杂度的问题

diff 策略

策略一：Web UI 中的 DOM 节点跨层级的移动特别少，可以忽略不记
策略二：拥有相同类的两个组件会产生类似的树形结构，拥有不同组件的会产生不同的树形结构
策略三：对于同一层级的一组子节点，可以通过唯一的 id 进行区分

tree diff

基于策略一，React 对树的算法进行了简洁明了的优化，对树进行分层比较，两个树只会对同一层级的节点进行比较。同一个父节点下的所有子节点，当发现节点不存在时，该节点及其子节点会被完全删除掉，不会进一步进行比较。这样只需要对树进行一次遍历，便可以完成整个 DOM 树的比较。

如下图所示，A 节点（包含其子节点）整个被移动到 D 节点下，由于 React 指挥简单的考虑同层级节点的位置交换，而对不同层级的节点，只有创建和删除操作。当根节点 A 消失了，就会直接销毁 A，当 D 发现多了一个子节点 A，会创建新的节点 A（包含子节点）作为其子节点.此时 diff 的执行情况为 createA->createB->creatB->Delete A。由此发现，当节点出现跨层级移动时，并不会出现移动节点的操作，这是一种影响 React 性能的操作，因此官方建议不要进行 DOM 节点进行跨层级的操作。

component diff

·如果是同一类型从组件，按照原策略继续比较 Virtual DOM 树即可
·如果不是，将该组件判断为 dirty component 从而替换整个组件下的所有子节点
·同一类型的组件，可能 virtual dom 没有任何变化，允许用户通过 shouldComponentUpdate()来判断该组件是否需要进行 diff 算法

element diff

当节点处于同一层级时，diff 提供三种节点操作，分别为INSERT_MARKUP(插入)，MOVE_EXISTING(移动)，REMOVE_NODE(删除)

  • INSERT_MARKUP：新的组件类型不在旧集合里，全新的节点，需要对新节点进行插入操作
  • MOVE_EXISTING：旧集合里有新组件的类型，且 element 是可更新的类型，可复用以前的 DOM 节点，进行移动操作
  • REMOVE_NODE：旧组件类型，在新集合里也有，但对应的 element 不同则不能直接服用和更新。或者旧组件不在新集合里。

如下图所示，旧集合中包含 A、B、C 和 D 更新后的集合包含 B、A、D 和 C 此时新旧集合进行 diff 差异化对比，发现 A!=B,则创建并插入 B 到新集合，删除旧集合 A，类推，创建并插入 A、D 和 C 删除 B、C 和 D。
针对这一现象，React 提出优化策略，允许开发者对同一层级的同组子节点，添加唯一的 key 进行区分，极大的提高了性能
带有 key 的节点进行 diff 差异化对比后，通过 key 发现新旧集合中的节点都是相同的节点，因此无需进行节点的删除和创建，只需要将旧集合中节点的位置进行移动操作就可以。
完成新集合所有节点的差异化对比后还是要对旧集合进行循环比哪里，判断是否存在新集合中没有但是旧集合中存在的节点。进行删除，到此 diff 操作全部完成。

diff 过程中的移动操作，也可能会影响性能。因此在开发过程中尽量减少将最后一个节点移动到列表首部的操作。这会造成该节点之前的节点进行移动，而非最后一个节点的移动。当节点数量过大或更新操作过于频繁时，在一定程度上会影响到 React 的渲染性能。

React Patch 方法

通过前面的内容，我们了解了 React 如何执行生命周期，更新 state，计算 tree diff 等，这一系列操作都还在 virtual dom 中进行的。React Pacth 实现了关键的最后一步。将 tree diff 计算出来的 DOM 差异队列更新到真实的 DOM 节点上。最终让浏览器能够渲染出更新的数据。这一步是通过遍历差异队列实现的。通过计算出全部差异并放入差异队列中，再一次性地执行 Patch 方法完成真是 DOM 的更新。

解密setState机制

发表于 2022-01-02 | 分类于 React

state 是 React 中重要的概念，React 是通过管理状态来实现对组件的管理。
众所周知，React 是通过 this.state 来访问 state 的，通过 setState 来更新 state。

绝对不要通过 this.state=1 这种操作来更新 state，这种操作不仅低效，而且很有可能被以后的操作抵消。

setState 异步更新

setState 循环调用的风险

当调用 setState 时,实际上会执行 enqueueSetState 方法,并对 partialState 以及_pendingStateQueue 更新队列进行合并操作。最终通过 enqueueUpdate 执行 state 更新。
而 performUpdateIfNecessary 方法会获取_pendingElement、_pendingStateQueue、_pendingForceUpdate,并调用 receiveComponent 和 updateComponent 进行组件更新。

如果在 shouldComponentUpdate 或 componentWillUpdate 方法中调用 setState，此时_pendingStateQueue!=null,则 performUpdateIfNecessary 方法会调用 updateComponent 方法进行更新，，但 updateComponent 又会调用 shouldComponentUpdate 和 componentWillUpdate 因此会造成循环调用，使得浏览器内存占满后崩溃。

React生命周期详解

发表于 2021-12-14 | 分类于 React

React 生命周期广义上分为三部分

  1. 挂载部分
    • constructor()
    • static getDerivedStateFromProps()
  2. 更新部分
    • static getDerivedStateFromProps()
    • shouldComponentUpdate()
    • getSnapshotBeforeUpdate()
    • ComponentDidUpdate()
  3. 卸载部分
    • ComponentWillUnmount()

React 的生命周期图

组件的挂载部分

constructor

如果不初始化 state 或不进行方法绑定，则不需要为 React 组件实现构造函数
换句话说通常在 React 中，构造函数仅用于以下情形

  1. 初始化 state
  2. 为事件处理函数绑定实例

static getDerivedStateFromProps(不常用)

见组件的更新部分/static getDerivedStateFromProps

组件的更新部分

static getDerivedStateFromProps(不常用)

getDerivedStateFromProps 方法会在 render 方法之前进行调用
并且在初始挂载以及后续更新都会被调用
此方法应返回需要更新的 state，若没有需要更新的 state，则返回 null
此方法仅适用于罕见的用例 即 state 的任何值都取决于 props

shouldComponentUpdate(不常用)

根据此方法的返回值，判断该组件的变化是否重新渲染
首次渲染或使用 forceUpdate() 时不会调用该方法。
官方建议此方法仅用来作为性能优化的方式而存在。后续版本，React 可能会将 shouldComponentUpdate 视为提示而不是严格的指令，并且，当返回 false 时，仍可能导致组件重新渲染。

getSnapshotBeforeUpdate()(不常用)

此方法在最近一次渲染输出之前进行调用，
他使得组件能在发生更改之前捕获一些信息(例如，滚动位置)
可将返回值传递给componentDidUpdate()的第三个参数
此方法并不常见

componentDidUpdate()

此方法会在更新后立刻进行调用,首次渲染不会执行此方法
主要用于组件更新后需要对 DOM 进行的操作
此方法允许 setState，但必须包裹在条件中，否则将会造成死循环。
此方法会导致额外的渲染

组件卸载部分

componentWillUnmount()

componentWillUnmount() 会在组件卸载及销毁之前直接调用。在此方法中执行必要的清理操作，例如，清除 timer，取消网络请求或清除在 componentDidMount() 中创建的订阅等。

componentWillUnmount() 中不应调用 setState()，因为该组件将永远不会重新渲染。组件实例卸载后，将永远不会再挂载它。

本篇文章记录《代码整洁之道》

有意义的命名

变量名要求是名副其实的，具有意义的并且需要其他使用者可以读出来的名称
类名和对象应该是名词或名词短语

函数

函数需要需要短小，并且保证一个函数只做一件事，无其他副作用
函数名称使用描述性语句，可以让用户清楚的知道这个函数做了什么

函数参数需要尽可能的少

try/catch 需要分离出来并且只处理一件事

注释

主张以好的代码替代注释，即观看函数名或变量便可知道该函数的意图

  • 法律信息
  • 提供信息的注释
  • 对意图的解释
  • 阐述
  • 警示
  • TODO 注释
  • 放大某种看起来不合理地方的重要性

格式
