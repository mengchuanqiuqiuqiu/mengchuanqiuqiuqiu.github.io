浅谈垃圾回收机制

发表于 2022-04-30 | 分类于 前端学习 |

程序运行过程中，几乎无时无刻不在对进程分配新的内存，但计算机的内存是有限的，没有办法无限制的分配新的内存，所以需要一个垃圾数据的回收机制来防止内存占满的问题。　　
对于Ｃ/Ｃ++等底层语言，内存的使用与释放是需要手动操作的。但对于 JS,python 等语言是通过垃圾回收器来处理不再使用的内存，通常不需要关心内存的使用状况。
但有时不当的代码会导致某些变量没有被垃圾回收器回收，积少成多造成内存泄漏，提高了应用卡顿的风险。

变量存储方式

JS 的变量分为原始类型与引用类型，不同变量的存储方式不同。原始变量存储在栈(Stack) 内存中，
引用类型存储在 堆(Heap) 中
当前存在原始类型的变量 a 与引用类型的变量 b，如图所示原始类型存在在栈中，而引用类型的变量 b 存储在堆中。

栈内存变量回收

栈内存的变量回收相对来说比较简单，当函数执行完毕后，函数的上下文从栈内存中弹出，存储在上下文中的变量也会被立即回收掉。对于上图中的变量 a 会被弹出，变量 b(对堆中的引用地址)也会被弹出，但变量 b 引用的对象{“C”:3}此刻仍然存在与堆中。
那么堆内存中的变量究竟该如何回收呢？

堆内存变量回收

标记清理

js 最常用的垃圾回收策略就是标记清理。当变量进入上下文，比如在函数内部声明一个变量时，这个变量会被加上存在于上下文中的标记。而在上下文中的变量从逻辑上讲永远不应该释放他们的内存，但变量离开上下文时，也会被加上离开上下文的标记。
垃圾回收程序运行的时候，会标记内存中存储的所有变量。然后它会将所有在上下文中的变量和上下文中的变量引用去掉。在此之后在被加上标记的变量就是待删除的了，原因是任何在上下文中的变量都访问不到他们了。
随后垃圾回收程序做一次内存清理销毁带标记的所有值并回收他们的内存。
虽然简单，缺点也十分的明显。多次回收操作后，会产生大量的内存碎片。如果需要申请一个较大的内存空间，哪怕内存足够也可能因为没有足够的连续内存而分配失败。

引用计数

另外一种没那么常用的方式是引用计数，其思路是对每个值都记录它被引用的次数。
声明一个变量并赋予他一个引用值 1，如果同一个值又被其他变量引用了，引用值则加 1，被覆盖则减 1。
当一个值的引用计数为 0 时，就说明无法再访问到这个值因此可以安全的回收其内存。
但引用计数会遇到循环引用的问题，导致内存无法正常清除，造成内存泄漏。

标记压缩

  1. 从一个 GC root 集合出发，标记所有活动对象
  2. 将所有活动对象移到内存的一端，集中到一起。
  3. 直接清理掉边界以外的内存，释放连续空间。

由于清理和整理的操作很麻烦，设置需要对堆进行多次搜索，故而堆越大耗时越多。

代际假设和分代收集

代际假说认为，大部分新对象的生存时间比较短，在一次垃圾回收周期内被回收。基于此，V8 将堆内存分为新生代和老生代。
所以可以将对象进行分代，从而对不同分代实施不同的垃圾回收算法，以达到更高的效率

总结

将内存占用量保持在一个较小的值可以让页面性能变得更好。优化内存占用的最佳手段就是保证在执行代码时只保存必要的数据。如果数据不再必要，可以把它设置成 null，从而释放其引用。这可以叫做解除引用
V8 垃圾回收器帮助 JS 使用者周期性的回收不再使用的内存。过多的对象会对垃圾回收器造成额外的负担，甚至造成页面的卡顿。　　
作为开发者应当有意识地减少全局变量的数量，及时移除不再使用的 DOM 对象、事件监听器以及计时器以减少垃圾回收器的负担。
